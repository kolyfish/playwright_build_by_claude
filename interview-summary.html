🐳
Docker
容器化測試環境
  • Base image：mcr.microsoft.com/playwright，內建瀏覽器，環境一致
  • Container 內裝 google-cloud-cli，具備直接上傳 GCS 能力
  • entrypoint.sh 處理：跑測試 → 上傳 → 回傳正確 exit code
  • 任何 CI 或本機都能一鍵執行，消滅「我這裡可以跑」問題
☁️
GCP — Cloud Storage
測試報告雲端託管
  • gsutil -m cp -r 並行上傳 HTML report 與 artifacts
  • 用 timestamp 做版本隔離，每次 run 獨立路徑
  • 報告 公開可讀，PM/QA 直接點 URL，不用進 CI
  • Service Account + Secrets，憑證不進 repo
⚙️
GitHub Actions
CI/CD 全自動流程
  • push 到 main / develop 自動觸發
  • Job Summary 直接顯示報告 URL，一眼看結果
  • 同時備份到 GitHub Artifacts（14 天保留）
  • Secrets 管理：GCP_SA_KEY、GCS_BUCKET
🧪
測試架構
Playwright + TypeScript + POM
  • Page Object Model：BasePage 封裝共用行為，LoginPage 繼承
  • E2E + API 測試分離，API 不開 browser 省資源
  • 失敗自動收集 截圖 + trace，debug 有跡可查
  • 無 hard sleep，全用 wait_for_selector / wait_for_load_state
面試時這樣說

「我用這個 side project 把整條 DevOps pipeline 跑通：本機寫測試 → Docker 封裝環境 → GitHub Actions 自動執行 → 報告上 GCS 讓所有人都能看到結果，不用進 Jenkins 或 CI 系統。」

💬 常見面試問答

問題 你的答案
為什麼用 Docker？ 環境一致性——Playwright 的瀏覽器版本在不同機器很難對齊，Docker 消滅這個問題
GCS 怎麼認證？ Service Account key 存在 GitHub Secrets，CI 時注入環境變數，不進 repo，符合最小權限原則
POM 的優點？ 頁面邏輯集中管理，locator 改動只改一處，測試本體維持乾淨
怎麼處理 flaky test？ 禁止 hard sleep，全用 wait_for_selector / wait_for_load_state，等事件不等時間
